# Atoms Demo 演示脚本

## 演示目标

用 5 到 8 分钟展示当前 demo 的核心闭环：

```text
query -> 意图识别 -> 分析/计划 -> 代码生成 -> 预览 -> 日志 -> 修复 -> 持久化
```

重点说明：这是一个类似 Atoms/Codex 的网页生成工作台原型，不是某个固定业务系统，也不是模板市场。

## 演示前准备

```bash
npm install
npm run dev
```

打开：

```text
http://localhost:3000
```

如果浏览器里已经有旧 session，可以清空站点数据，或直接点击“新项目”开始新 session。

## 演示流程

### 1. 展示首屏入口

说明点：

- 左侧是 session 工作空间列表。
- “新项目”相当于新建一个空 session。
- 空 session 不会进入历史列表。
- 中间只保留输入框、上传按钮和发送按钮，避免把注意力从 query 上移开。

### 2. 咨询类 query

输入：

```text
帮我分析一下需求管理系统应该先做哪些功能
```

预期表现：

- 页面创建一个新 session。
- 左侧出现来自第一条消息的 session 标题。
- 中间展示用户消息和 assistant 分析。
- 结果包含问题理解、分析建议、下一步。
- 不出现右侧生成工作区。

解说：

```text
这个请求只是让系统做需求分析，不是让系统直接生成应用，所以它不会进入代码生成和预览流程。
```

### 3. 实现类 query

点击“新项目”，输入：

```text
帮我做一个校园社团活动报名系统，需要活动发布、学生报名、名额限制和签到核销
```

预期表现：

- 系统识别为实现类请求。
- 中间展示需求理解、功能拆解、页面结构、数据结构、任务步骤、预期文件。
- 右侧出现生成工作区。
- 预览标签展示可运行页面。
- 代码标签可以查看 `index.html`、`styles.css`、`app.js`。
- 执行日志展示从意图识别到预览渲染的过程。

解说：

```text
这里展示的是核心生成闭环：系统不是套固定业务模板，而是从 query 里提取主题和功能点，再生成页面、文件和预览。
```

### 4. 修改类 query

在同一个 session 继续输入：

```text
把页面改成移动端优先，并增加筛选功能
```

预期表现：

- 系统识别为修改类请求。
- 中间展示变更理解、影响范围、任务步骤和预期生成文件。
- 右侧工作区更新为新的生成结果。
- 执行日志包含修改和重新检查过程。

解说：

```text
session 就是工作空间。后续修改会沿用当前上下文，而不是重新生成一段无关内容。
```

### 5. 附件类 query

点击上传按钮，选择一个需求说明文件，再输入：

```text
根据附件帮我实现这个页面
```

预期表现：

- 输入区和消息里展示附件名。
- 分析结果中体现附件已作为上下文。
- 实现类流程继续生成计划、文件和预览。

解说：

```text
当前阶段先把附件名纳入上下文，验证交互链路。后续可以在这个位置扩展真实文档解析。
```

### 6. 代码、日志和修复入口

切换右侧工作区：

- 预览：展示生成页面。
- 代码：展示生成文件内容。
- 执行日志：解释系统做了哪些步骤。

如果检查失败，页面会显示失败原因和“一键修复”按钮。点击后应更新生成结果并记录修复日志。

### 7. 持久化验证

刷新页面。

预期表现：

- 左侧 session 仍然存在。
- 点击 session 后，消息、附件名、分析、生成结果和当前工作区标签仍可恢复。

解说：

```text
当前用 localStorage 保存完整 session，先保证演示闭环稳定可复现。
```

## 推荐演示顺序

1. 咨询类 query：证明不会误入生成。
2. 实现类 query：证明能生成计划、代码和预览。
3. 修改类 query：证明同一个 session 支持继续迭代。
4. 刷新页面：证明本地持久化可用。

## 当前边界

- 当前不依赖外部服务，便于本地稳定演示。
- 当前附件只纳入文件名上下文，还没有解析文件正文。
- 当前生成结果是本地确定性生成，用于证明产品闭环和交互方向。
- 当前不展示固定业务模板，避免偏离“根据用户需求生成应用”的目标。
