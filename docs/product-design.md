# Atoms Demo Product Design

## 1. 产品定位

Atoms Demo 是一个网页版 AI 应用生成工作台。用户输入想法或问题后，系统需要理解用户意图，完成需求分析、实施规划、代码生成，并把生成的应用以可视化网页形式展示出来。

这个产品不是固定业务模板生成器。用户的 query 是唯一入口，不同想法应该生成不同分析、不同计划、不同文件和不同预览。

## 2. 要解决的问题

用户经常只有一个模糊想法，例如“做一个需求管理系统”或“帮我分析这个功能应该怎么设计”。真正的难点不是展示一段静态说明，而是把想法推进为可运行、可检查、可继续修改的产品原型。

本 demo 要解决的是：

- 把模糊想法拆成清晰需求。
- 区分“咨询分析”和“实际实现”。
- 让用户看到智能体正在做什么。
- 生成可查看的代码和可运行的网页预览。
- 支持用户在同一个 session 中继续追问和修改。

## 3. 核心原则

### Query 驱动，而不是模板驱动

系统不应该让用户从 CRM、游戏、预约、看板等固定模板中选择。用户输入什么，系统就围绕这个输入做理解、计划和生成。

### Session 就是工作空间

每个 session 都是一块独立工作空间，包含对话、附件、分析结果、生成计划、生成文件、预览状态和后续修改记录。

点击“新建对话”只进入一个空白工作状态。如果用户没有发送内容，就不应该新增 session。

### 过程必须可见

用户发送 query 后，需要能看到系统进入了哪一步：

1. 意图识别。
2. 需求分析。
3. 实施计划。
4. 代码生成。
5. 预览渲染。
6. 校验与修复建议。

### 首屏保持极简

首屏应该像 Atoms/Codex 一样，把注意力放在输入上：

- 左侧只放核心导航和 session。
- 中间是主要输入区。
- 输入框旁只保留上传和发送。
- 不在首屏堆复杂面板。

### 结果必须可体验

实现类请求最终不能只返回文字。它应该产出可查看的文件、可运行的预览，以及能继续要求修改的上下文。

## 4. 主要用户流程

### 咨询类请求

示例：

```text
帮我分析一下需求管理系统应该先做哪些功能
```

期望流程：

1. 创建或继续当前 session。
2. 判断为咨询类请求。
3. 输出问题理解、分析建议、下一步。
4. 不展示构建状态、文件列表或预览。
5. 用户确认要实现后，再进入实现流程。

### 实现类请求

示例：

```text
帮我做一个校园社团活动报名系统，需要活动发布、学生报名、名额限制和签到核销
```

期望流程：

1. 创建或继续当前 session。
2. 判断为实现类请求。
3. 提取产品主题和功能点。
4. 生成需求规格、页面结构、数据结构、任务步骤和预期文件。
5. 进入构建状态。
6. 后续生成代码文件并渲染预览。

### 带附件请求

期望流程：

1. 附件作为当前 session 的上下文。
2. 分析结果需要明确体现附件已被纳入。
3. 后续支持解析需求文档、截图或接口说明。
4. 生成结果要结合附件内容。

### 迭代修改请求

示例：

```text
把页面改成移动端优先，并增加筛选功能
```

期望流程：

1. 继续使用当前 session 和当前 artifact。
2. 判断这是修改请求。
3. 更新计划和相关文件。
4. 重新渲染预览。
5. 展示变更摘要。

## 5. 当前闭环范围

当前只围绕核心体验闭环推进，不在设计文档里扩展远期架构。

闭环顺序：

1. 完善首页交互。
2. 完善意图识别。
3. 展示实施计划。
4. 展示生成结果区域。
5. 根据 query 生成可运行页面。
6. 展示预览、代码和执行日志。
7. 支持错误提示和一键修复。
8. 保存 session、消息、附件名、分析结果和生成结果。
9. 整理最终演示说明。

## 6. 页面结构

### 左侧 Sidebar

作用：管理工作空间和历史 session。

包含：

- Atoms 标识。
- 新建对话。
- 资源入口。
- 用户创建的 session 历史。

规则：

- 历史列表来自用户实际创建的 session。
- 空 session 不进入列表。
- session 标题默认来自第一条用户输入。

### 中间 Conversation

作用：承载输入、对话、分析结果和生成过程。

包含：

- 初始标题“你今天想创造什么？”。
- 用户消息。
- assistant 分析结果。
- 实现类任务的构建状态。

规则：

- 咨询类请求只展示分析。
- 实现类请求展示计划和构建状态。
- 不展示和当前 query 无关的固定内容。

### 生成工作区

作用：后续展示生成文件、代码和预览。

目标结构：

- 文件列表。
- 代码查看区。
- 预览区。
- 运行或校验状态。
- 修改后的版本记录。

## 7. 当前核心闭环

当前阶段只围绕一条闭环建设：

```text
用户 query
-> 意图识别
-> 咨询分析或实现计划
-> 生成结果区域
-> 代码生成
-> iframe 预览
-> 错误检查和一键修复
-> localStorage 持久化
```

这个闭环跑通之前，不扩展与当前演示无关的远期能力。

## 8. 技术架构

### 当前架构

- `src/components/build-room/BuildRoom.tsx`：页面 shell、session 状态、输入、上传入口和结果展示。
- `src/features/query/analyzeUserQuery.ts`：纯函数 query 分析。
- `src/features/query/analyzeUserQuery.test.ts`：意图识别测试。
- `src/app/globals.css`：全局布局和样式。

当前状态存在 React state 和 `localStorage` 中。这个阶段优先验证交互和产品方向。

### 下一步模块

下一步只新增与核心闭环直接相关的模块：

- `GenerationWorkspace`：承载预览、代码和执行日志。
- `generateApp`：根据 query 生成本地静态页面文件。
- `PreviewFrame`：用 sandbox iframe 渲染生成页面。
- `CodeViewer`：展示生成文件和代码。
- `RunLog`：展示分析、生成、检查和修复过程。
- `sessionStorage`：用 localStorage 保存 session、消息、附件名、分析结果和生成结果。

## 9. 持久化边界

当前只使用 localStorage。保存内容包括：

- session。
- 消息。
- 附件名。
- 分析结果。
- 生成结果。
- 当前工作区标签状态。

后续是否扩展持久化方式，等核心闭环跑通后再单独讨论。

## 10. 成功标准

评审者打开 demo 后应该能确认：

- 这是一个 AI 应用生成工作台，而不是普通业务系统。
- 可以从输入框开始一个新的 session。
- 系统能区分咨询和实现。
- 实现类请求能生成清晰计划。
- 后续可以自然扩展到代码生成和预览。
